# Q9 — `git clone` vs `git fetch` vs `git pull`

## 📖 Overview

Git provides three related but distinct commands for getting data from a
remote repository. Understanding the difference is essential for safe,
predictable collaboration.

---

## 1. `git clone`

`git clone` creates a **complete local copy** of a remote repository,
including its full commit history and all branches. It is typically run
**once**, at the very start of a project, to set up your local working copy.

```bash
git clone https://github.com/your-username/iot-summer-school-2026.git
```

What it does:
- Downloads the entire repository (`.git` history + working files)
- Automatically creates a remote named `origin`
- Checks out the default branch (usually `main`) into a working directory

---

## 2. `git fetch`

`git fetch` downloads new commits, branches, and tags from the remote
repository **without** modifying your current working directory or local
branches. It updates your local copy of the remote-tracking branches
(e.g. `origin/main`) so you can review changes before merging them in.

```bash
git fetch origin
```

What it does:
- Updates `origin/main`, `origin/dev`, etc. with the latest remote state
- Does **not** touch your local `main`, `dev`, etc.
- Safe to run at any time — it never rewrites your working files

---

## 3. `git pull`

`git pull` is effectively **`git fetch` + `git merge`** (or `git rebase`,
depending on configuration) in a single step. It fetches remote changes
**and immediately merges them** into your current local branch.

```bash
git pull origin main
```

What it does:
- Fetches the latest commits from `origin`
- Merges those commits into your currently checked-out branch
- Can create merge commits or conflicts if local and remote history diverge

---

## 🔍 Comparison Table

| Command       | Downloads History | Modifies Local Files | Typical Use Case                          | Risk Level |
|---------------|:------------------:|:---------------------:|--------------------------------------------|:----------:|
| `git clone`   | ✅ (full repo)      | ✅ (creates new dir)   | First-time setup of a repository           | 🟢 Low      |
| `git fetch`   | ✅ (updates refs)   | ❌                     | Checking for remote updates safely         | 🟢 Low      |
| `git pull`    | ✅ (via fetch)      | ✅ (merges into branch)| Syncing your current branch with remote    | 🟡 Medium   |

---

## 💡 Examples

**Cloning the project for the first time:**
```bash
git clone https://github.com/your-username/iot-summer-school-2026.git
cd iot-summer-school-2026
```

**Checking what changed on the remote before merging anything:**
```bash
git fetch origin
git log main..origin/main --oneline
```

**Pulling the latest `dev` branch changes into your local `dev`:**
```bash
git checkout dev
git pull origin dev
```

---

## ✅ Best Practices

1. **Prefer `fetch` + review + `merge`** over blind `pull` on shared branches
   like `main`, especially in a team setting — it lets you inspect incoming
   changes first.
2. **Always `pull` before starting new work** on a branch to minimize merge
   conflicts.
3. **Use `git pull --rebase`** on feature branches to keep history linear
   and avoid unnecessary merge commits.
4. **Never `pull` with uncommitted local changes** you care about — commit
   or stash them first (`git stash`).
5. **Clone once, fetch/pull often** — there is rarely a reason to re-clone
   a repository you already have locally.
